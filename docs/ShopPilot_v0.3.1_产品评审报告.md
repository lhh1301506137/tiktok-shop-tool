## 一、代码架构 — 做对了什么

### ✅ 亮点

1. **MAIN/ISOLATED 双脚本架构**：`injected.ts` 在 MAIN world 拦截 fetch/XHR，`index.ts` 在 ISOLATED world 做 chrome 通信和 UI 注入。这是 MV3 下最正确的数据捕获方案。

2. **Shadow DOM 隔离**：FAB 按钮、Track 按钮全部封装在 closed Shadow DOM 中，完全不受 TikTok 页面 CSS 影响。

3. **虚拟滚动 + 存储分片**：`useVirtualList` hook 只渲染可见的 ~15 张卡片（而非 2000+），配合 `creators_0/1/2...` 的 200 条一块的分片存储，解决了 chrome.storage.local 的 1MB 限制。

4. **多策略 DOM 选择器**：7+ 个 CSS 选择器降级路径 + TreeWalker 文本模式匹配作为兜底，对 TikTok 频繁改版有一定抵抗力。

5. **AI 服务加固**：30s 超时 + 自动重试 + 指数退避 + 友好错误提示，生产级可靠。

## 二、代码风险 — 必须修的问题

### 🔴 P1：批量邀约队列在内存中，Service Worker 重启会丢失

**问题**：`inviteQueue.ts` 的 `queueState` 是一个全局变量。MV3 的 Service Worker 会在空闲 30 秒后被终止。如果用户发起 50 人批量邀约，Worker 在第 20 个时被杀掉，队列状态和进度全部丢失，用户看到的是"无响应"。

**影响**：用户丢失进度，可能重复邀约同一批达人。

**修复方案**：将 `queueState` 持久化到 `chrome.storage.local`，Worker 重启时自动恢复。添加"恢复中断的批量任务"功能。

---

### 🔴 P1：Usage 计数的并发竞态

**问题**：`storage.ts` 的 `incrementUsage()` 执行的是 read → increment → write 三步操作，不是原子的。在批量邀约模式下，多个快速连续的 `incrementUsage` 调用可能读到同一个旧值，导致计数丢失。

**影响**：用户可以绕过每日限额。

**修复方案**：
- 方案 A：在 background 中用队列串行化所有 increment 操作
- 方案 B：使用 `chrome.storage.session` 或内存锁来保证原子性

---

### 🔴 P1：竞品监控的 Tab 爆炸风险

**问题**：`refreshAllTrackedProducts` 为每个产品创建一个隐藏 tab 来抓取数据。50 个产品 = 50 次 tab 创建/销毁，每次间隔 2-5 秒，总耗时约 4 分钟。

**影响**：
- 资源消耗巨大，用户可能注意到浏览器变慢
- Service Worker 可能在刷新过程中被终止
- Chrome 对频繁创建 tab 可能有限流

**修复方案**：复用单个"工作 tab"依次导航，或并行 3-5 个 tab 批量处理。

### 🟡 P2：补充问题

## 三、产品策略评审

### 🎯 产品-市场匹配度：**高**

TikTok Shop 是当前增长最快的电商平台。达人邀约是卖家增长的核心引擎，但 Seller Center 的原生界面极其笨拙，手动操作效率低下。ShopPilot 解决的是一个**真实的、高频的痛点**。

**目标用户画像**：
- 中小型 TikTok Shop 卖家
- 独立站 dropshipper
- 没有预算用 Kalodata/FastMoss 这类高端 SaaS 的人
- 东南亚市场的跨境卖家

### ⚠️ 关键功能缺口

| 缺失功能 | 重要性 | 说明 |
|:---|:---:|:---|
| **模板管理** | 高 | 很多卖家有经过验证的邀约模板，希望 A/B 测试而非每次 AI 生成 |
| **多店铺支持** | 高 | 代运营/小团队通常管理 5-10 个店铺 |
| **达人回复追踪** | 高 | 邀约发出后无法跟踪结果，数据闭环断裂 |
| **团队协作** | 中 | Business 层级应支持多人共享达人库和邀约记录 |
| **数据导出增强** | 中 | CSV 导出需要更加健壮，支持 CRM 导入格式 |

### 🏗️ 已建 vs 应建

- ✅ **达人抓取 + AI 邀约 + 批量发送**：这是核心杀手功能，优先级正确
- ⚠️ **Listing 优化**：锦上添花，不是核心痛点，但能增加付费感知价值
- ⚠️ **竞品监控**：同上，但实现成本高（tab 爆炸问题），ROI 需要重新评估

## 四、商业模式评审

### BYOK 模式：双刃剑

| 优势 | 劣势 |
|:---|:---|
| 零 LLM 成本，永不亏钱 | **80% 的电商卖家不知道什么是 API Key** |
| 数据 100% 本地，隐私极强 | 首次配置耗时长，churn 风险极高 |
| 用户可选便宜模型（DeepSeek） | 无法追踪用户的 AI 使用模式来优化产品 |

**核心矛盾**：你的目标用户是电商卖家，不是开发者。BYOK 模式对技术用户友好，但会成为普通卖家的 **#1 流失原因**。

### 定价分析

| 层级 | 价格 | 每日邀约 | 评价 |
|:---|:---:|:---:|:---|
| Free | $0 | 5 | **太少**。5 个邀约约等于 3 分钟的使用时间，用户来不及感受到价值就被锁住了 |
| Pro | $19/月 | 100 | 定价合理，是"不需要审批"的小额商业支出 |
| Business | $49/月 | 无限 | **价值不足**。和 Pro 的唯一区别是"无限"，缺乏专属功能来支撑 $30 的差价 |

## 五、增长与竞争

### 竞争护城河：**薄**

当前的护城河几乎为零：
- 代码全在客户端，容易被逆向
- BYOK 模式意味着你甚至不拥有"AI 大脑"
- 没有网络效应，没有数据飞轮

**建立护城河的方向**：
1. **更优的 Prompt 工程**：经过 A/B 测试验证的高转化率邀约提示词
2. **深度工作流集成**：让产品如此深入 Seller Center 流程，切换成本极高
3. **社区 + 知识库**：建立达人质量评分体系等独有数据资产

### 增长机制：**缺失**

- ❌ 没有 referral / 邀请机制
- ❌ 没有分享或社交传播触发点
- ❌ 没有 SEO 内容策略（博客、教程等）
- ✅ 落地页质量高，但流量从哪来？

---

## 六、合规风险

### TikTok ToS 风险：⚠️ 中等

这是**最大的外部风险**。TikTok 对自动化操作的态度日趋严格。

- **批量邀约**的 3-8 秒随机延迟是好的，但激进使用仍可能触发风控
- **MAIN world fetch 拦截**是最隐蔽的数据获取方式，但本质仍是未授权的数据抓取
- **缓解措施**：在产品内加入明确的使用指南和风险提示；让用户自己控制速率

### Chrome Web Store 合规：✅ 良好

- 权限最小化，有完整的审核说明文档
- 无混淆代码，无远程脚本加载
- 隐私政策完整且准确

### GDPR：✅ 优秀

无服务器 = 零数据责任。所有达人数据都在用户本地浏览器中。

## 八、总结

ShopPilot 在**技术实现上已经达到了上架水平**——架构设计成熟、性能优化到位、安全意识强。但在**商业就绪度上还有明显差距**：

1. **BYOK 模式是最大的双刃剑**：它保护了你的利润但杀死了你的转化率
2. **三个 P1 级技术债**（队列持久化、计数竞态、tab 管理）需要在积累真实用户前修复
3. **增长引擎为零**：好产品不会自己卖出去，需要 referral、内容、社区至少一个飞轮
4. **护城河薄弱**：需要尽快建立用户粘性和数据资产

**建议路径**：先修 3 个 P1 bug → 上架获取早期用户反馈 → 基于反馈决定是否增加托管 AI 选项 → 建立增长机制。

不要追求"完美再上线"。当前版本已经能解决真实痛点，早日上架、收集数据、快速迭代才是正确策略。
